Provider Tutorials

Everything you need to create, publish, and test your own routers.

1. Create Your First Router
Create a permissioned, modular router with unified chat and search endpoints. This tutorial will walk you through the process from start to finish.
  1. Install SyftBox: Download from syftbox.net (desktop or CLI). Follow the installation instructions for your platform.
  2. Add Your App: In SyftUI, click Add App and enter your app’s GitHub URL, or use the CLI: syftbox add-app <github-url>. SyftBox will clone your app, recognize run.sh, and start your router automatically.
  3. Monitor Startup: Watch the logs in SyftUI or CLI. Once started, your router will appear in the app list with a dashboard link.
  4. Access the Dashboard: Click the dashboard link in SyftUI to open your router’s web interface. You’ll see onboarding and can start configuring endpoints.
Tip: Use clear, kebab-case names (e.g., customer-support-chat) for easy identification.

Error Response Format

If something goes wrong, the router will return a structured error response.

{
  "error": {
    "code": "ROUTER_NOT_FOUND",
    "message": "Router 'my-router' not found",
    "details": {
      "router_name": "my-router",
      "available_routers": ["router1", "router2"]
    }
  }
}

2. Publishing and Monetization
Learn how to publish your router, set pricing, and make it discoverable to clients in the federated marketplace.
  1. Open SyftUI or CLI: Select your app in the SyftBox dashboard or use the CLI.
  2. Publish: Click Publish in the UI or run syftbox publish-app <app-name> --summary "..." --description "..." --tags "..." --pricing "..." in the CLI.
  3. Fill Metadata: Add a clear summary, description, and relevant tags. Set pricing for each endpoint (per request, per token, etc.).
  4. Confirm: Complete the publish process. Your router will now be discoverable by clients.
Tip: Use competitive pricing and clear descriptions to attract users. Update your metadata as your service evolves.

3. Test your router
Verify your router’s endpoints and monitor its status to ensure reliability and performance.
  1. Status & Health: In SyftBox UI or CLI, check the Status and Health indicators for your app: syftbox status <app-name>.
  2. Monitor Usage: Track usage and revenue in SyftBox UI or via CLI analytics.
  3. API Testing: Use the built-in OpenAPI docs, Swagger UI, cURL, or Postman to test endpoints (chat/search) and validate responses.
  4. Troubleshooting: If you encounter issues, review logs in SyftUI or CLI and check error responses for details.
Tip: Regularly test endpoints and monitor logs to catch issues early and maintain high service quality.

4. [Advanced] Custom Routers

What is a custom router?
A custom router lets you define your own logic, endpoints, and data flows beyond the default templates. This is ideal for advanced use cases, integrations, or when you want to expose new types of services.
Which endpoints can I define?
  • Chat Endpoint (/chat): Accepts a model name and message history, returns an AI-generated response. Implement in chat_service.py.
  • Search Endpoint (/search): Accepts a query and options, returns search results or RAG completions. Implement in search_service.py.
  • Health Endpoint (/health): Returns status and metadata for monitoring. Implement in spawn_services.py or a dedicated health service.
  • Custom Endpoints: You can add any additional endpoints (e.g., analytics, feedback, custom data processing) by creating new service files and registering them in your router config.
How to structure your service files:
  • Each endpoint should have a dedicated service file (e.g., chat_service.py for chat, search_service.py for search).
  • Define request and response schemas using Pydantic models for type safety and OpenAPI docs.
  • Implement business logic in service classes or functions, following FastAPI best practices.
  • Register your endpoints in the router's main configuration (e.g., config.py).
Best practices:
  • Use clear, versioned endpoint paths (e.g., /v1/chat).
  • Document all endpoints and schemas with docstrings and OpenAPI annotations.
  • Handle errors gracefully and return structured error responses (see example above).
  • Validate all inputs and sanitize user data to prevent security issues.
  • Keep business logic modular for easier testing and extension.
  • Test endpoints using SyftBox UI, CLI, or tools like Swagger UI and Postman.
Extensibility & Security Tips:
  • Leverage dependency injection for authentication, logging, and monitoring.
  • Restrict access to sensitive endpoints using permissions and roles.
  • Log important events and errors for observability.
  • Keep your dependencies up to date and review third-party packages for security.
Example: Minimal Custom Chat Endpoint 
# chat_service.py
from fastapi import APIRouter
from pydantic import BaseModel

router = APIRouter()

class ChatRequest(BaseModel):
    model: str
    messages: list[dict]

class ChatResponse(BaseModel):
    message: dict

@router.post("/chat", response_model=ChatResponse)
def chat_endpoint(request: ChatRequest):
    # Your custom logic here
    reply = {"role": "assistant", "content": "Hello from your custom router!"}
    return ChatResponse(message=reply)
This example shows a minimal FastAPI chat endpoint. You can expand the logic, add authentication, or connect to your own models as needed.
  1. When creating a router, select Custom type for a template with TODOs.
  2. Edit the generated service files (chat_service.py, search_service.py, etc.) to implement your own logic and endpoints as described above.
  3. Update pyproject.toml and config.py as needed to register new endpoints and dependencies.
  4. Test and validate your router using OpenAPI docs, SyftBox UI, CLI, or API calls.
Tip: Custom routers are ideal for advanced use cases, integrations, and when you want full control over your API surface.

Client Tutorials

How to discover, chat, and integrate with published routers.

5. Discover routers and chat
Find published routers and use chat endpoints.
  1. Switch to Client Mode: In SyftBox UI, click the Switch to Client Mode button or use the CLI command syftbox client.
  2. Browse Routers: In the client dashboard, you can browse published routers, filter by service type, tags, or price.
  3. Select and Use: Select a router and use the chat endpoint directly in the UI. You can also interact via the CLI or API.
Tip: You can combine multiple data sources for richer context in chat.

6. Use the API
Integrate published routers into your applications using the REST API.
  1. Find API Docs: Access the OpenAPI documentation (Swagger UI, cURL, or directly via API endpoint) for the published router you want to integrate.
  2. Test Endpoints: Use cURL, Python, JavaScript, or your preferred client to test endpoints (chat/search) and validate responses.
  3. Integrate: Integrate the endpoints into your application, respecting owner-set permissions and pricing.
Tip: Always check the API docs for required parameters and authentication details.

SDK Integration Examples

Here are ready-to-use code snippets for integrating with published routers using Python or JavaScript.

Python 

import requests

# Chat with router
response = requests.post(
    "http://localhost:8080/chat",
    json={
        "model": "llama3",
        "messages": [
            {"role": "user", "content": "Hello!"}
        ]
    }
)

result = response.json()
print(result["data"]["message"]["content"])

JavaScript 

// Chat with router
const response = await fetch(
  "http://localhost:8080/chat",
  {
    method: "POST",
    headers: {
      "Content-Type": "application/json"
    },
    body: JSON.stringify({
      model: "llama3",
      messages: [
        { role: "user", content: "Hello!" }
      ]
    })
  }
);

const result = await response.json();
console.log(result.data.message.content);
